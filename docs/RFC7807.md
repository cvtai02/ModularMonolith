# RFC 7807 — Problem Details for HTTP APIs

## Overview

[RFC 7807](https://www.rfc-editor.org/rfc/rfc7807) defines a standard error format (`application/problem+json`) for HTTP APIs to carry machine-readable details about errors in a response, avoiding the need to define new error response formats for every API.

---

## Problem Details Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | URI | No | A URI identifying the problem type. When dereferenced, it should provide human-readable documentation. Defaults to `about:blank`. |
| `title` | string | No | A short, human-readable summary of the problem type. Should not change between occurrences. |
| `status` | integer | No | The HTTP status code. Must match the actual response status code. |
| `detail` | string | No | A human-readable explanation specific to this occurrence of the problem. |
| `instance` | URI | No | A URI identifying the specific occurrence. May or may not be dereferenceable. |

Extensions (additional members) are allowed.

---

## Media Type

```
Content-Type: application/problem+json
```

---

## Example

### Request
```http
GET /orders/42 HTTP/1.1
Host: api.example.com
```

### Response
```http
HTTP/1.1 404 Not Found
Content-Type: application/problem+json

{
  "type": "https://example.com/problems/order-not-found",
  "title": "Order Not Found",
  "status": 404,
  "detail": "Order with ID 42 does not exist.",
  "instance": "/orders/42"
}
```

---

## Validation Error Extension Example

RFC 7807 allows extending the problem object with domain-specific fields. A common extension for validation errors:

```json
{
  "type": "https://example.com/problems/validation-error",
  "title": "One or more validation errors occurred.",
  "status": 400,
  "detail": "See 'errors' for details.",
  "instance": "/orders",
  "errors": {
    "quantity": ["must be greater than 0"],
    "productId": ["is required"]
  }
}
```

---

## Rules & Constraints

- `type` **SHOULD** be an absolute URI. If absent, consumers must treat it as `about:blank`.
- `status` **SHOULD** match the HTTP response status code. It is informational; the actual status code governs behaviour.
- `title` **SHOULD NOT** change between occurrences of the same `type`. It identifies the problem class, not the instance.
- `detail` is for the specific occurrence — it may change per request.
- Problem types **MUST** be documented at their `type` URI.
- Extensions **MUST NOT** redefine existing members.

## Usage in this Project (CakeTeaCofi)

All API error responses follow RFC 7807. The `type` URIs are namespaced under each module:

| Module | Type Prefix |
|--------|-------------|
| Catalog | `https://caketacofi.com/problems/catalog/` |
| Products | `https://caketacofi.com/problems/products/` |

### Common Problem Types

| Type | Status | Title |
|------|--------|-------|
| `…/validation-error` | 400 | One or more validation errors occurred. |
| `…/not-found` | 404 | The requested resource was not found. |
| `…/conflict` | 409 | The request conflicts with the current state. |
| `…/unprocessable` | 422 | The request was well-formed but contains semantic errors. |
